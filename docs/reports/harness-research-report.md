# 工业前端 Harness 系统研究报告

## 1. 结论先行

Harness 不是脚手架，也不只是 CI/CD。

它是一套把前端研发、测试、发布、运行、回滚、审计与安全治理组织成工业化生产线的系统。在工业前端场景中，Harness 的价值不在于“更快起项目”，而在于让多团队、多终端、多现场环境下的交付能力变得：

- 可复制
- 可观测
- 可回滚
- 可审计
- 可治理

## 2. 为什么工业前端必须建设 Harness

工业前端的典型约束包括：

- 多终端：大屏、工控机、操作台、平板、手持设备
- 多环境：办公网、生产网、DMZ、边缘节点、离线环境
- 强集成：MES / SCADA / ERP / IoT / SSO / 网关 / CMDB
- 长生命周期：系统需要维护 5–10 年甚至更久
- 强合规：审计、最小权限、网络白名单、发布审批
- 高可用：很多页面不能接受“上线后再慢慢修”

## 3. Harness 的定义与边界

### 3.1 定义

Harness 是工业前端的生产装配系统（Assembly Harness）：通过标准化的任务模型、配置契约、执行环境、治理策略和观测机制，将工程最佳实践沉淀为可组合、可重复、可审计的交付能力。

### 3.2 负责范围

- 项目骨架与模板
- 数据接入标准
- 权限接入标准
- 测试门禁
- 构建发布
- 运行观测
- 回滚能力
- 审批与审计
- 版本治理
- AI/自动化执行治理

### 3.3 不负责范围

- 具体业务需求实现
- 每个项目的业务规则细节
- 替代后端平台或设备平台
- 重写现有监控、CI、制品库本身

## 4. 与相关系统的关系

| 对象 | 主要职责 | 与 Harness 的关系 |
|---|---|---|
| 脚手架 | 起项目、生成初始代码 | 只是 Harness 的一个子能力，不等于 Harness |
| CI/CD | 执行流水线 | Harness 定义流水线标准、门禁、证据、回滚和观测 |
| 测试平台 | 执行测试、展示报告 | Harness 决定哪些测试必须跑、何时阻断、如何纳入治理 |
| Design System | 视觉规范与组件资产 | Harness 负责将其版本化、门禁化、升级化 |
| 研发门户 | 入口、表单、可视化 | 门户只是外壳，Harness 才是生产底座 |
| AI / ACP Harness | 代码生成、补测试、自动化执行 | 应被纳入 Harness 作为受控 Runner/Task，而不是平行体系 |

## 5. 成熟 Harness 的五大域

### 5.1 能力域
- 模板与工程骨架
- 权限/数据/配置 SDK
- 组件与图形能力
- 多终端适配
- 离线/弱网能力
- 浏览器自动化与回归能力

### 5.2 治理域
- 版本策略
- 兼容矩阵
- 升级助手
- 质量门禁
- 变更审批
- 生命周期管理

### 5.3 接口域
- CLI / API
- Task / Policy / Recipe Spec
- App Manifest
- Environment Config
- Artifact Spec
- Telemetry Schema

### 5.4 观测域
- 构建耗时
- 测试通过率
- 发布频次
- 回滚率
- 异常率 / 白屏率 / 性能预算
- 版本与线上行为关联

### 5.5 安全域
- 最小权限
- Token 分层
- Secrets 不落盘
- 审批与审计
- 供应链安全
- 制品签名 / 哈希 / SBOM

## 6. 推荐设计模式

- Golden Path
- 插件化
- 契约优先
- 策略即代码
- 环境即代码
- 制品优先
- 可观测默认开启

## 7. 高风险反模式

- 把脚手架当 Harness
- 只做门户，不做底座
- 平台强绑架，业务只能绕开
- 没有版本治理和迁移路径
- 只有发布，没有运行观测
- 忽略弱网、旧内核、离线和白名单环境
- AI 直接绕过 PR 和治理规则

## 8. 推荐落地路线

### 阶段一：MVP
- 项目模板
- Task / Recipe / Policy / Telemetry 最小模型
- CLI 入口
- Git + CI + Playwright 基础集成
- canary + rollback 路径
- 样板项目 1–2 个

### 阶段二：标准化
- 兼容矩阵
- 版本治理
- 升级助手
- 契约测试
- 环境治理
- 分级质量门禁
- SBOM / 制品哈希 / 签名

### 阶段三：平台化
- Portal
- 资产目录
- 多租户 / 多组织支持
- 策略即代码集中治理
- 发布编排与批次管理
- AI 受控接入体系

## 9. 最终结论

如果要在工业前端里真正组建一套 Harness 系统，第一优先级不是做一个漂亮的平台首页，也不是先统一组件库，而是先统一：

- 标准任务模型
- 标准配置契约
- 标准门禁
- 标准证据
- 标准回滚
- 标准观测
